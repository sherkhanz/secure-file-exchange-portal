# Threat Model: Secure File Exchange Portal

**Version:** 1.0  
**Framework:** STRIDE

---

## 1. Description of the System

Secure File Exchange Portal is an API-first service for temporary file sharing. It runs as a single-container FastAPI service where authenticated users can upload files and generate expiring download tokens. Recipients use tokens to download files with no further authentication. All activity is emitted as structured stdout log lines. State is persisted in a SQLite database on a Docker-managed named volume.

```
[ Internet / Curl Client ]
          │  HTTPS (future) / HTTP (current)
          ▼
  ┌───────────────────────┐
  │   Docker Bridge Net   │
  │  ┌─────────────────┐  │
  │  │  sfep_api :8000 │  │
  │  │   (FastAPI)     │  │
  │  └──────┬──────────┘  │
  │         │             │
  │  ┌──────▼──────────┐  │
  │  │  Named Volumes  │  │
  │  │  sfep_db        │  │   ← SQLite: portal.db
  │  │  sfep_files     │  │   ← Raw uploaded files
  │  └─────────────────┘  │
  └───────────────────────┘
```

---

## 2. Scope of the Document

This threat model covers the core architecture of the Secure File Exchange Portal. The analysis focuses on the FastAPI application container, its SQLite persistence layer, local file storage volumes, and the Docker Compose infrastructure. All threats are evaluated directly against the main.py and docker-compose.yml source files.

## 3. Assets

| Asset | Description | Sensitivity |
|-------|-------------|-------------|
| Download tokens | UUID4 tokens stored in `links` table, grant unauthenticated file access | Critical |
| Uploaded files | Raw file content stored on `sfep_files` volume | High |
| API token | Static secret in `docker-compose.yml`, gates all protected endpoints | High |
| SQLite database | `portal.db` on `sfep_db` volume - file metadata, link state, expiry data | High |
| Audit log stream | Stdout log lines captured by Docker log driver | Medium |
| Container runtime | FastAPI process running as `portaluser` (UID 1001) inside Docker | Medium |

## 4. Threat Vectors

### Threat 1 - Broken Access Control on `/download/{token}` (IDOR)

| Field | Detail |
|-------|--------|
| **STRIDE Category** | **Elevation of Privilege** |
| **Affected Asset** | Download tokens, Uploaded files |
| **Affected Endpoint** | `GET /download/{token}` |
| **Impact** | **CRITICAL** |

**Scenario**

The `/download/{token}` endpoint has no `Depends(require_auth)` guard. Any unauthenticated HTTP client that can reach port 8000 may attempt a download. The token is a UUID4, which is generated by Python's `uuid.uuid4()` using the OS PRNG. Because there is no rate-limiting or failed-attempt lockout anywhere in the codebase, an attacker can issue sustained enumeration requests against `/download/<uuid>`.

```python
# main.py
@app.get("/download/{token}")
def download_file(token: str):
    ...
```

Furthermore, a token is never invalidated after a single successful download. Any party who intercepts or receives a forwarded token URL can download the file an unlimited number of times until it expires or is manually revoked - with no audit differentiation between the intended recipient and a third party.

### Threat 2 - Credential Exposure via Static Hardcoded API Token

| Field | Detail |
|-------|--------|
| **STRIDE Category** | **Spoofing** |
| **Affected Asset** | API token, Audit log stream |
| **Affected Component** | `docker-compose.yml` → `API_TOKEN` |
| **Impact** | **HIGH** |

**Scenario**

The entire access-control plane of the application (upload, link creation, and revocation) is gated on a single static secret that is hardcoded in plaintext in `docker-compose.yml`. This creates several risks:

1. **Version control exposure:** If `docker-compose.yml` is committed to a repository, the token is permanently embedded in git history.
2. **No per-user identity:** There is one token for all callers. There is no way to revoke access for a single compromised user without rotating the credential for every user simultaneously.
3. **Audit non-attribution:** Because `require_auth` only validates the token string without resolving it to a specific identity, the actual user behind the request is not recorded in the audit logs.

```python
# main.py
def require_auth(x_api_token: Optional[str] = Header(default=None)):
    if x_api_token != API_TOKEN:
        raise HTTPException(status_code=401, ...)
```

### Threat 3 - Unrestricted File Upload

| Field | Detail |
|-------|--------|
| **STRIDE Category** | **Tampering** |
| **Affected Asset** | Uploaded files, Container runtime |
| **Affected Endpoint** | `POST /upload` |
| **Impact** | **HIGH** |

**Scenario**

The upload handler reads the entire file body and writes it directly to the filesystem with no content inspection of any kind. The stored filename is constructed as `{uuid}_{original_filename}`, meaning the original extension is preserved on disk:

```python
stored_name = f"{file_id}_{file.filename}"
dest        = os.path.join(STORAGE_PATH, stored_name)
with open(dest, "wb") as fh:
    fh.write(contents)
```

This creates two attack paths:

1. **Future misconfiguration:** If a reverse proxy is ever placed in front of the container and `STORAGE_PATH` is inadvertently mapped to a web-accessible directory, a `.php` or `.cgi` upload becomes remote code execution.
2. **Malware distribution:** An authenticated attacker can use the portal as a distribution channel for malware, ransomware, or exfiltration payloads, served via unauthenticated `/download` links.

### Threat 4 - Audit Log Repudiation

| Field | Detail |
|-------|--------|
| **STRIDE Category** | **Repudiation** |
| **Affected Asset** | Audit log stream |
| **Affected Component** | `audit()` function in `main.py` |
| **Impact** | **MEDIUM** |

**Scenario**

All security-relevant events are emitted exclusively to stdout via Python's `logging` module. Docker captures this stream into its internal log buffer, which:

1. **Rotates without archival** - an attacker who triggers sufficient log volume can cause earlier evidence to rotate out.
2. **Is mutable by a root-level host actor** - the Docker host's root user can truncate or delete log files at `/var/lib/docker/containers/<id>/<id>-json.log` with no detection.
3. **Contains no tamper-evident chain** - there is no HMAC, sequence number, or hash chain linking log entries.

### Threat 5 - Sensitive Data Exposure via Unencrypted Transport and Volume Storage

| Field | Detail |
|-------|--------|
| **STRIDE Category** | **Information Disclosure** |
| **Affected Asset** | Uploaded files, API token, SQLite database |
| **Affected Components** | Docker port binding (`0.0.0.0:8000`). Named volumes `sfep_db`, `sfep_files` |
| **Impact** | **HIGH** |

**Scenario**

The service binds to `0.0.0.0:8000` and serves all traffic over plain HTTP with no TLS termination. This means:

1. **The `x-api-token` header is transmitted in cleartext** on every authenticated request. Any network-level observer can capture the credential with a passive packet capture.
2. **File content is transmitted in cleartext** - the body of every `/download/{token}` response is unencrypted in transit.
3. **Named volumes `sfep_db` and `sfep_files` are stored unencrypted** on the Docker host at `/var/lib/docker/volumes/`. Any process running as root on the host has direct access to all uploaded files and the full SQLite database.

```yaml
# docker-compose.yml
ports:
  - "8000:8000"
volumes:
  sfep_db:
  sfep_files:
```

## 5. Risk Prioritization

| # | Threat | STRIDE | Impact | Likelihood | Blast Radius | Priority | Mitigation |
|---|--------|--------|--------|------------|--------------|----------|------------|
| 1 | Unauthenticated file download / IDOR | Elevation of Privilege | Critical | High | Any uploaded file retrievable by anyone with network access and a guessed token | P0 | 1. Add `Depends(require_auth)` to `/download/{token}`<br>2. Implement `max_downloads` counter in links table<br>3. Add `slowapi` rate limiting at 5 req/min per IP |
| 2 | Hardcoded static API token | Spoofing | High | High | Full write access to upload, link creation, and revocation for any party who reads the compose file | **P0** | 1. Move `API_TOKEN` to Docker Secret<br>2. Issue per-caller hashed API keys<br> 3. Propagate `user_id` to all audit events |
| 3 | Unrestricted file upload | Tampering | High | Medium | Portal becomes a malware staging and distribution platform; RCE possible under future proxy misconfiguration | **P1** | 1. Add MIME-type allowlist<br>2. Use `python-magic` for byte-level validation<br> 3. Integrate ClamAV sidecar |
| 4 | Non-repudiable stdout-only audit log | Repudiation | Medium | High | Complete audit trail erasable by root in one command; no actor attribution survives incident review | **P1** | 1. Persist events to append-only SQLite `audit_log` table<br> 2. Forward logs to external SIEM<br> 3. Add HMAC chain per entry |
| 5 | Cleartext transport and unencrypted volumes | Information Disclosure | High | Medium | API token and all file content exposed to any network observer; entire database accessible on host compromise | **P1** | 1. Add nginx TLS sidecar<br> 2. Restrict API to internal Docker network<br> 3. Enable volume encryption via LUKS |

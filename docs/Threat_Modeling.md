# Threat Modeling: Secure File Exchange Portal

## 1. Description of the System

Secure File Exchange Portal is an API-first service for temporary file sharing. It runs as a single-container FastAPI service where authenticated users can upload files and generate expiring download tokens. Recipients use tokens to download files. All activity is emitted as structured stdout log lines. State is persisted in a SQLite database on a Docker-managed named volume.

## 2. Scope of the Document

This threat model covers the core architecture of the Secure File Exchange Portal. The analysis focuses on the FastAPI application container, its SQLite persistence layer, local file storage volumes, and the Docker Compose infrastructure. All threats are evaluated directly against the main.py and docker-compose.yml source files.

---

## 3. Assets

| Asset | Description | Sensitivity |
|-------|-------------|-------------|
| Download tokens | UUID4 tokens stored in `links` table, grant file access | Critical |
| Uploaded files | Raw file content stored on `sfep_files` volume | High |
| API token | Secret loaded from `.env` file, gates all protected endpoints | High |
| SQLite database | `portal.db` on `sfep_db` volume - file metadata, link state, expiry data | High |
| Audit log stream | Stdout log lines captured by Docker log driver | Medium |
| Container runtime | FastAPI process running as `portaluser` (UID 1001) inside Docker | Medium |

## 4. Threat Vectors

### Threat 1 - Broken Access Control on `/download/{token}` (IDOR)

| Field | Detail |
|-------|--------|
| **STRIDE Category** | Elevation of Privilege |
| **Affected Asset** | Download tokens, Uploaded files |
| **Affected Endpoint** | `GET /download/{token}` |
| **Impact** | Critical |
| **Status** | Remediated |

**Scenario**

The `/download/{token}` endpoint previously had no `Depends(require_auth)` guard. Any unauthenticated HTTP client that could reach port 8000 could attempt a download. The token is a UUID4, generated by Python's `uuid.uuid4()` using the OS PRNG. Without rate-limiting or failed-attempt lockout, an attacker could issue sustained enumeration requests against `/download/<uuid>`.

Furthermore, a token was never invalidated after a single successful download. Any party who intercepted or received a forwarded token URL could download the file an unlimited number of times until expiry or manual revocation.

**Remediation Implemented**

- `Depends(require_auth)` added to `GET /download/{token}` - endpoint now requires a valid API token
- `slowapi` rate limiting implemented at 5 requests per minute per IP on `/download`

---

### Threat 2 - Credential Exposure via Static Hardcoded API Token

| Field | Detail |
|-------|--------|
| **STRIDE Category** | Spoofing |
| **Affected Asset** | API token, Audit log stream |
| **Affected Component** | `docker-compose.yml` ŌåÆ `API_TOKEN` |
| **Impact** | High |
| **Status** | Remediated |

**Scenario**

The entire access-control plane of the application (upload, link creation, and revocation) was gated on a single static secret hardcoded in plaintext in `docker-compose.yml`. This created several risks:

1. **Version control exposure:** If `docker-compose.yml` was committed to a repository, the token was permanently embedded in git history.
2. **No per-user identity:** There is one token for all callers. There is no way to revoke access for a single compromised user without rotating the credential for every user simultaneously.
3. **Audit non-attribution:** Because `require_auth` only validates the token string without resolving it to a specific identity, the actual user behind the request is not recorded in the audit logs.

**Remediation Implemented**

- `API_TOKEN` removed from `docker-compose.yml`
- Token now loaded from `.env` file via `python-dotenv`
- `.env` added to `.gitignore` - token no longer committed to version control

### Threat 3 - Unrestricted File Upload

| Field | Detail |
|-------|--------|
| **STRIDE Category** | Tampering |
| **Affected Asset** | Uploaded files, Container runtime |
| **Affected Endpoint** | `POST /upload` |
| **Impact** | High |
| **Status** | Remediated |

**Scenario**

The upload handler previously read the entire file body and wrote it directly to the filesystem with no content inspection. The stored filename was constructed as `{uuid}_{original_filename}`, preserving the original extension on disk. This created two attack paths:

1. **Future misconfiguration:** If a reverse proxy was placed in front of the container and `STORAGE_PATH` was inadvertently mapped to a web-accessible directory, a `.php` or `.cgi` upload would become remote code execution.
2. **Malware distribution:** An authenticated attacker could use the portal as a distribution channel for malware, ransomware, or exfiltration payloads.

**Remediation Implemented**

- `ALLOWED_EXTENSIONS` set implemented in `main.py` - only `.txt`, `.pdf`, `.png`, `.jpg`, `.jpeg`, `.docx`, `.xlsx`, `.csv` permitted
- Extension check runs on every upload - rejected files return HTTP 422
- `slowapi` rate limiting implemented at 5 requests per minute per IP on `/upload`

### Threat 4 - Audit Log Repudiation

| Field | Detail |
|-------|--------|
| **STRIDE Category** | Repudiation |
| **Affected Asset** | Audit log stream |
| **Affected Component** | `audit()` function in `main.py` |
| **Impact** | Medium |
| **Status** | Open |

**Scenario**

All security-relevant events are emitted exclusively to stdout via Python's `logging` module. Docker captures this stream into its internal log buffer, which:

1. **Rotates without archival** - an attacker who triggers sufficient log volume can cause earlier evidence to rotate out.
2. **Is mutable by a root-level host actor** - the Docker host's root user can truncate or delete log files at `/var/lib/docker/containers/<id>/<id>-json.log` with no detection.
3. **Contains no tamper-evident chain** - there is no HMAC, sequence number, or hash chain linking log entries.

### Threat 5 - Sensitive Data Exposure via Unencrypted Transport and Volume Storage

| Field | Detail |
|-------|--------|
| **STRIDE Category** | Information Disclosure |
| **Affected Asset** | Uploaded files, API token, SQLite database |
| **Affected Components** | Docker port binding (`0.0.0.0:8000`). Named volumes `sfep_db`, `sfep_files` |
| **Impact** | High |
| **Status** | Open |

**Scenario**

The service binds to `0.0.0.0:8000` and serves all traffic over plain HTTP with no TLS termination. This means:

1. **The `x-api-token` header is transmitted in cleartext** on every authenticated request. Any network-level observer can capture the credential with a passive packet capture.
2. **File content is transmitted in cleartext** - the body of every `/download/{token}` response is unencrypted in transit.
3. **Named volumes `sfep_db` and `sfep_files` are stored unencrypted** on the Docker host at `/var/lib/docker/volumes/`. Any process running as root on the host has direct access to all uploaded files and the full SQLite database.

```yaml
# docker-compose.yml
ports:
  - "8000:8000"
volumes:
  sfep_db:
  sfep_files:
```
